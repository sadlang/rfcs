# ملخّص

إنشاء إضافة VS Code جديدة باسم **`sadlang-vscode`** (مفتوحة المصدر **Apache-2.0**، مستودع
متعدّد الحِزَم بـ npm workspaces) تكون **عميل LSP رفيعًا** يعيد استخدام خادم اللغة الموجود
`sad-lsp-server` بدل تكرار ذكاء اللغة، مع دعم RTL والتمييز النحويّ وتشغيل/بناء برامج `.ص`. ومعها
حزمة عقد توسّع عامّة موثَّقة **`@sadlang/vscode-api`** تتيح لإضافات مرافِقة التوسّع عبر **نقاط
محايدة** (تشغيل، ذكاء، تنقيح…) — على نموذج منصّة VS Code نفسها. هذا القلب الرفيع يحلّ محلّ نموذج
أوّليّ متراكم غير منشور، ببنية نظيفة وحدّ واضح.

# الدافع (Motivation)

- **مصدر حقيقة واحد للذكاء**: مستودع اللغة يحوي خادم LSP أصليًّا ناضجًا (`sad-lsp-server` فوق
  `sad_core` = المعجميّ/النحويّ/AST). يجب أن تستهلكه الإضافة لا أن تكرّره؛ التكرار يُنتج انحرافًا
  (تحذيرات «غير معرّف» كاذبة، إكمال متخلّف عن القواعد الحقيقيّة).
- **هويّة عربيّة (RTL)**: تجربة تحرير عربيّة أولى ممتازة هي بوّابة تبنّي اللغة.
- **قابليّة توسّع**: عقد عامّ موثَّق يفتح نظامًا بيئيًّا من إضافات مرافِقة (أوّل/ثالث طرف) تبني فوق
  «ص» كما تبني الإضافات فوق VS Code.
- **نظافة معماريّة**: بنية حِزَم واضحة بدل نموذج أوّليّ متضخّم، تسهّل الصيانة والمساهمة.

# الشرح التوجيهي (Guide-level explanation)

يثبّت المطوّر إضافة **«ص»** فيحصل فورًا على: إكمال ذكيّ، تشخيصات آنيّة، الانتقال للتعريف والمراجع
وإعادة التسمية، رموز المستند، التنسيق، الطيّ، التمييز النحويّ العربيّ، **لوحة RTL**، وأوامر
**تشغيل/بناء** عبر أدوات اللغة (`sad-run`/`sad-build`)، وتنقيح أساسيّ (DAP).

```sad
دالة جمع(أ، ب)
    ارجع أ + ب
نهاية
```
عند الكتابة يُكمِل المحرّر `جمع(` ويعرض توقيع الدالة، والتشخيصات تأتي من خادم اللغة مباشرةً (لا
قاعدة ثابتة منفصلة).

**للإضافات المرافِقة**: يحصل المطوّر على واجهة الإضافة عبر التفاوض المُصدَّر:
```
const api = vscode.extensions.getExtension("sadlang.sadlang-vscode")?.exports?.getApi(1);
api.runTargets.register({ id: "...", label: "...", run: ... });
```
نقاط التوسّع **محايدة الاسم** (تشغيل، مزوّدو ذكاء، محوّلات تنقيح، عروض، مركز أوامر)، فأيّ إضافة
مرافِقة تُسجِّل قدراتها دون أن يحرسها القلب.

# الشرح المرجعي (Reference-level explanation)

- **مصدر الحقيقة (`language-truth/`)**: نعم — **تُولَّد بيانات الإكمال/المدمجات آليًّا من
  `language-truth`** (لا تهريد يدويّ) وتُغذّى بها مزوّدات `sad-lsp-server`. هذا **بوّابة عدم
  تراجع**: الإضافة لا تُشحَن حتى يتفوّق ذكاء LSP على أيّ قاعدة ثابتة مؤقّتة (تبقى الأخيرة كـ
  fallback دون اتصال فقط). يُعالَج توصيل المدمجات الناقص في خادم اللغة، لا في الإضافة.
- **المحلل/المفسّر/المترجم**: لا تغيير في اللغة. التنقيح المتقدّم يعتمد على دعم `--debug-server`
  في المفسّر (تبعيّة تُتعقَّب في مستودع اللغة).
- **الأدوات (LSP/المنسّق)**: **إعادة استخدام** `sad-lsp-server` (عميل رفيع يكتشفه ويشغّله) +
  CLI `sad-run`/`sad-build`. لا إعادة بناء لذكاء اللغة في الإضافة.
- **عقد `@sadlang/vscode-api`**: حزمة أنواع عامّة موثَّقة. **هجين**: للقدرات القياسيّة تُستعمل
  **نقاط VS Code الرسميّة** (`InlineCompletionItemProvider`، `vscode.chat`/`vscode.lm`،
  `DebugAdapter`، `vscode.authentication`)؛ ولِما يخصّ «ص» يُصدَّر كائن typed عبر `activate()`
  بدالة تفاوض `getApi(n)` (نمط إضافة Git المدمجة `getAPI(1)`). السياسة: **semver + إضافيّ
  (append) غالبًا + دعم N-1 بنوافذ إهمال**. العقد **مفتوح**: أيّ إضافة مرافِقة تُسجِّل عبر السوكِت،
  **والقلب لا يحرس الوصول** (نقاط توسّع فقط). نقاط مبدئيّة محايدة: `runTargets`، `lspClient`،
  `rtlHost`، `commandCenter`، `statusBar`، `events`.
- **البنية**: `npm workspaces` — حِزَم `client` (مضيف الإضافة) · `lsp-client` (اكتشاف/تشغيل
  `sad-lsp-server`) · `grammar` (tmLanguage/snippets/themes) · `ui-rtl` · `features` (باقة UX
  منتقاة). خادم LSP **يبقى في مستودع اللغة** (لا يُنسَخ هنا).
- **النشر**: **Marketplace + Open VSX**؛ تتبّع **opt-in فقط** يحترم `telemetry.telemetryLevel`.
- **التوافق الخلفي**: النموذج الأوّليّ السابق **غير منشور** على أيّ سجلّ ⇒ لا كسر لمستخدمين.
  ترقيم **جديد 0.x ← 1.0** عند نضج MVP.

# السلبيات (Drawbacks)

- جهد إعادة بناء القلب رفيعًا بدل ترحيل النموذج الأوّليّ كما هو.
- اعتماد على نضج توصيل LSP من SoT — يُدار ببوّابة عدم تراجع + fallback مؤقّت.
- صيانة عقد عامّ مُصدَّر بالإصدارات (عبء semver).

# المبرّرات والبدائل (Rationale and alternatives)

- **لماذا إعادة بناء رفيع لا ترحيل المونوليث؟** الترحيل ينقل التكرار (قاعدة إكمال ثابتة تنافس
  LSP) والدَّين التقنيّ؛ القشرة النظيفة تعطي حدًّا واضحًا وتستهلك مصدر الحقيقة الواحد.
- **لماذا Apache-2.0؟** منح براءات صريح + حماية علامة + تبنٍّ كامل، أنسب لمشروع مدعوم مؤسّسيًّا
  من ترخيص بلا منح براءات.
- **لماذا عقد هجين لا API ضخم خاصّ؟** الاتّكاء على نقاط VS Code الرسميّة يقلّل السطح ويزيد
  التوافق؛ والعقد الرفيع الخاصّ يغطّي ما يخصّ «ص» فقط بإصدارات مستقرّة.
- **أثر عدم الفعل**: بقاء نموذج أوّليّ متضخّم يكرّر LSP ويصعب صيانته وتوسيعه.

# أعمال سابقة (Prior art)

- **VS Code**: منصّته API **عامّة موثَّقة مُصدَّرة**؛ إضافة Git المدمجة تَعرِض `getAPI(1)` عامًّا
  تستهلكه إضافات أخرى — النموذج المتبنّى هنا.
- **Copilot**: يبني على نقاط VS Code الرسميّة (`InlineCompletionItemProvider`، `vscode.lm`/
  `vscode.chat`) لا على API بَيْنيّ خاصّ — يؤكّد نهج «استعمل النقاط الرسميّة حيث وُجدت».
- نظام إضافات مفتوح فوق منصّة موثَّقة درس مُجرَّب في محرّرات عدّة.

# أسئلة غير محسومة (Unresolved questions)

- السطح الدقيق لـ`@sadlang/vscode-api` (أسماء/تواقيع النقاط) — يُحسم أثناء التنفيذ مع أوّل مستهلك.
- أيّ ميزات UX بالضبط تدخل نطاق MVP الأوّل (الباقة المنتقاة) مقابل المؤجَّل.
- آليّة سدّ فجوات توصيل المدمجات في خادم اللغة (RFC منفصل في `tools/` للغة/LSP).

# إمكانات مستقبلية (Future possibilities)

- إضافات مرافِقة (أوّل/ثالث طرف) تبني فوق العقد العامّ وتُوسّع «ص» بقدرات متخصّصة.
- طبقة أدوات معياريّة مستقبليّة (مثل MCP) لربط أدوات «ص» بنماذج/وكلاء بشكل host-agnostic.
- دعم محرّرات أخرى تتحدّث LSP عبر نفس `sad-lsp-server`.
